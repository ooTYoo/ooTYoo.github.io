RSA oracle Attack

发表于 2018-07-02 更新于 2022-03-17 分类于 crypto

这篇笔记介绍在密码学中著名的Bleicherbacher Orcale攻击。Bleicherbacher Orcale攻击针对RSA加密及签名方案中，使用固定填充方式带来的实现中的安全问题。为解决这类问题，后续推荐使用带随机性的填充方式如PSS和OAEP来代替传统的PKCS填充。

1998年密码学家Daniel Bleichenbacher发现在SSL协议中[1]，使用PKCS#1 v1.5填充方式的实现，会面临Adaptive-chosen ciphertext攻击。攻击者可以通过将一系列精心选择的密文发送给SSL协议server，通过分析server的不同相应，在不获取私钥的情况下，可以破解目标密文对应的明文。

在2006年，Bleichenbacher发布了对于PKCS#1 v1.5填充方式的RSA签名算法，当padding解析存在缺陷的情况下，对于很小的公钥e=3，可以通过
简单的计算获取对任意数据的签名，该漏洞存在在openssl实现中。

本文包含攻击原理解释，以及对应的python代码实现。内容保含如下典型RSA Oracle攻击方法

  • Parity Oracle Attack
  • Bleichenbacher attack on Encryption(98)
  • ROBOT (Return Of Bleichenbacher oracle aTtack)

什么是Oracle

在client-server通信模型中，假设server端存储有一秘密信息，client端为获取该密码信息，向server发送特定请求，server端对client请求进行简单的回复，例如回复1bit的值（0或1）。如果client端通过分析server的回复信息，可以推测出server的密钥，则server端的回复行为可以被称为某种Oracle。

一个最简单的例子是猜数字游戏。假设server端的秘密是0-99中的某一个数字（记为key），client端任选一个数字x，发送询问“x>=key?"，在server只回复是（1）或否（0）的情况下，client端通过多次询问，即可最终确定key的值。

Parity Oracle Attack

Parity oracle指当攻击者输入由RSA公钥(e,N)加密的密文C后，服务器端返回明文P是否为偶数的信息。该Oracle泄漏明文P最后1bit信息（LSB，Least Significant Bit），攻击者可通过选择密文攻击，推测出P的内容。

Bleichenbacher attack on Encryption(98)

Bleichenbacher在其98年的论文 Chosen Ciphertext Attacks Against Protocols Based on the RSA Encryption Standard PKCS #1 中阐明，当使用PKCS#1_5方式对RSA加密明文进行填充时，存在padding oracle attack。

server端解密时计算步骤为

  1. 使用私钥d解密密文，得到中间结果 m′=ctdmod  Nm'=ct^d \mod Nm′=ctdmodN
  2. 判断m’是否符合PKCS#1填充规则，如果 m’高位两个byte不为0x0002,则直接报错

Bleichenbacher Singature Forgery(06)

Bleichenbacher’06 attack 是一种可以利用实现漏洞可以对公钥 e=3e=3e=3 直接伪造其签名的攻击方式。

PKCS#1定义的签名算法如下

  1. 对于待签名消息m计算对应hash值hash(m)
  2. 做填充EM=00||01||FF…FF||00||ASN.1(hash)||hash(m))
  3. 对EM进行签名 S=EMdmod  NS=EM^d \mod NS=EMdmodN

对于固定填充的消息，其签名相当于计算有限域上第3步结果的e次方根。虽然直接计算精确结果难度较大，但对于 e=3e=3e=3的场景，对某些特殊格式的EM，可以有效计算出其立方根，即该消息对应的签名。这里假设服务器端的校验函数存在实现缺陷，签名校验时能通过如下形式的填充

EM′=Sdmod  N=00∣∣01∣∣FF...FF∣∣00∣∣ASN.1(hash)∣∣hash(m))∣∣garbageEM' = S^d \mod N = 00||01||FF...FF||00||ASN.1(hash)||hash(m)) || garbage EM′=SdmodN=00∣∣01∣∣FF...FF∣∣00∣∣ASN.1(hash)∣∣hash(m))∣∣garbage

此处garbage部分被服务器抛弃，当garbage之前的部分符合pkcs签名填充方式时，服务器认为其中的hash值即为合法的hash值。Note 中记录了如何通过手算即可伪造合法的签名。以hash算法为sha-1, RSA公钥长度为3072bit为例，做如下安排

标记D为二进制串00||ASN.1(hash)||hash(m))对应的大整数，该部分共36Byte即288bit。garbage对应的大整数记为G。
通过调整m，可以使得D满足

C=2288−D,Cmod  3=0C = 2^{288} - D, C \mod 3 = 0 C=2288−D,Cmod3=0

结论及建议

虽然Bleichenbacher Oracle攻击发现距今已有20多年，由于协议更新换代较慢，在很多实现中任然存留该安全漏洞。这一类的典型是2017年在facebook服务器上爆出的ROBOT攻击（Return of Bleichenbacher Oracle Threat）[5]。

从预防角度，尽量使用带随机性的加密/签名方案，如OAEP/PSS方案，另外对于公钥e，推荐选取65537，应避免使用 e=3e=3e=3 的小指数。
