0x1 编码的基本问题

简单讲，所谓编码就是通讯过程中（如两方Alice/Bob通信），Alice将一串原始信息M变换为另一种形式C，将C通过特定信道传递到通信的对端，Bob接收到C‘后可以从中恢复得到通信的内容M。编码可以是为了尽量提升信息传递的效率，也可以是为了保证通信的可靠性。考虑如下场景，当通信信道较差时，如向太阳系外飞去的水手号将其观测信息跨过茫茫太空回传给地球，C在信道中传递不可避免得会产生错误，此时可以通过添加冗余信息来保护原始信息。但冗余又会降低通信效率，由此引入第一个关键问题

SPHINCS

发表于 2022-10-02 更新于 2022-10-19

SPHINCS是基于hash的后量子签名算法，该算法在NIST第三轮中胜出。该算法和XMSS算法最大的不同在于其是stateless的，算法中无须维护可用密钥对的指标。其算法原理和基本组件和XMSS高度相同。本文中对该算法作个总结性介绍。

0x1 基本组件

SPHINCS算法是基于hash单向性的，其算法包括如下基本组件

  1. tweakable hash 算法，在标准的hash算法基础上，添加了key和消息掩码
  2. hash address scheme，标准中定义了5中不同的地址类型、
  3. WOTS+ 单次签名算法
  4. XMSS Merkel tree
  5. Hyper Tree
  6. FORS， Forestt Of Random Set

tweakable hash算法在单向函数中增加了key和掩码，提升了SPHINCS算法的安全行，也是WOTS+中加号的来源。对应的函数为

Tl:Bn×B32×Bln→Bnmd←Tl(PK.seed,ADRS,Mesg)T_l :\mathbb{B}^n \times \mathbb{B}^{32} \times \mathbb{B}^{ln} \rightarrow \mathbb{B}^n \\ md \leftarrow T_l(PK.seed, ADRS, Mesg) Tl​:Bn×B32×Bln→Bnmd←Tl​(PK.seed,ADRS,Mesg)

SPHINCS中的32Byte地址结构定义和XMSS具有较大不同，SPHINCS定义了五种不同类型的地址

  1. WOTS_HASH，用于WOTS+的hash中
  2. WOTS_PK，用于WOTS+公钥的压缩算符中
  3. TREE，用于Merkel Tree的构造中
  4. FORS_TREE，用于FORS Tree的构造中
  5. FORS_ROOTS，FORS Tree的root构造中

WOTS+的构造过程和XMSS相同，使用SK.seed 通过 PRF(SK.seed, ADRS) 产生 len 个 n-Byte的随机序列，之后使用单项函数迭代w-1次产生公钥。和XMSS中不同的是公钥的压缩方式。在XMSS中，len 个公钥使用L-Tree 生成一个n-byte公钥，在SPHINCS中，使用tweakable hash一次产生pk=T_len(PK.seed, wotspkADRS, wotspk[0:len-1]) 。

WOTS+的签名包含了len 个n-byte的hash计算结果。在SPHINCS中WOTS+用于在Hypter-Tree中，对于layer-0的root做签名。

Merkel Tree组件中，构建一颗高度为h 的平衡二叉树，共有 2h2^h2h 个叶节点，每个节点对应一个压缩后的WOTS+公钥。树的根节点作为XMSS的公钥。

XMSS的签名包含1个n-byte的WOTS+签名，以及由h个n-byte 串组成的Auth_path。共 (h+len)n(h+len)n(h+len)n Byte。验签过程需重现root节点的计算过程。

Hyper-Tree(HT)高度为 h′h'h′， 包含了 ddd 层，每层包含一系列高度为 h′/dh'/dh′/d 的Merkerl Tree，这些子树具备相同的密码学参数n,w 。SPHINCS中HT的叶结点为FORS Tree的公钥。layer-0为FORS-Tree，其上均为XMSS-Tree，layer-(n-1) 用于对其 layer-n 中子树的root进行WOTS+签名，所有签名中包含了d个WOTS+签名，总长度为 $d(h’/d + len)n $ Byte。

0x4 算法优缺点

算法的优点包括：

  1. 良好的侧信道防御能力
  2. 极少的安全假设，只以来于单向函数的单向性
  3. 安全性分析较简单
  4. 复用了大量现有的密码学组建，方便构建

缺点：

  1. 调用大量的单项函数计算，性能慢
  2. 签名尺寸太大

在实际实现时，可以通过空间换性能对第一个缺点做缓解，但第二个缺点是制约SPHINCS+被广泛使用的关键因素。也是NIST发起第四轮评审的主要考虑，在第四轮中，主要是基于编码的密码算法了。

0x3 Merkle’s Tree Scheme

OTS方案中，每个公私钥对只能使用一次，如果使用OTS作为签名方案，需要派发大量的公钥。1979年Merkle提出了一种方案来解决大量公钥的问题，该方案中的核心组件是称为Merkle‘s Tree的结构。

Merkle‘s Tree是完全二叉树。假设现在有 2h2^h2h 组OTS公私钥对，其Merkle‘s Tree性质如下

  • 叶子节点对应于每个OTS公钥y的hash

0x4 MSS方案

基于hash构建的签名方案具备以下特点

  • 依赖于较少的安全假设
  • 具备抵抗量子计算机攻击的能力
  • 实现方式较为简单
  • 具备自然的抗SCA分析能力等

基于hash的PQC算法可以分为两大类

  1. Stateful的，需要在使用中维护状态信息，如XMSS, LMS
  2. Stateless的，如NIST PQC候选算法SPHINCS

0x4.1 W-OTS+方案

WOTS+在WOTS的基础上，将底层的单项函数替换为代key的hash算法，并且将hash计算的输入进行了加掩码保护，hash key和掩码通过PRF(pseudo Random function) 产生，PRF的输入包括

  1. n-byte的seed
  2. 32Byte的Address结构

在后续的WOTS签名和验签的基础上，都添加了基于掩码的保护逻辑

0x4.2 L-Tree

RFC定义了对多个W-OTS+公钥进行压缩时使用的L-Tree算法，该算法可以将多个(个数不要求为2k2^k2k)个chain末端的pk压缩为单个L-Tree的根节点。并使用L-Tree构建Merkle Tree。

0x4.3 Multi-Tree

XMSS方案中需要保障每个W-OTS+公私钥对只使用一次，在签名方案中需要维护当前可用的签名密钥对信息。当使用单个Merkle Tree支持多个公私钥对时，Merkle-Tree根节点的计算需要所有密钥对的多次hash计算，这给密钥派生操作带来了巨大的负担。Multi-Tree方案可以缓解该问题。

所谓的Multi-Tree，是使用多个具备相同参数（高度，w位宽等）的Merkle-Tree组成的hyper Tree，该hyperTree的叶子节点为使用WOTS+方案对对应Merkle-Tree根节点的签名。HyperTree简化了公钥真实行的校验所需的hash计算量。

0x1 多变量公钥密码算法（MPKC）

Rainbow算法属于多变量公钥密码算法（MPKC, Multivariate Public Key Cryptography）的一种。MPKC基于的困难问题是“在有限域上求解多变量非线性多项式组”。MPKC中所用的多项式为二次多项式（有研究表明更高次数多项式反而会降低系统安全性）。MPKC中通常定义m个多项式 p(1)…p(m)p^{(1)}\ldots p^{(m)}p(1)…p(m), 其中每个多项式为n个变量 $x_1, x_2,\ldots, x_n $ 的二次多项式。

其中所有的系数和变量取值都在有限域 Fq\mathbb{F}_qFq​ 上。 当为n个变量 $ x_i$ 输入特定的值之后，这m个多项式求值可以得到 m个结果。

P=(y1,y2,…,ym)=(p(1)(x1,…,xn),…,p(m)(x1,…,xn))\mathcal{P} = (y_1,y_2,\ldots,y_m) = (p^{(1)}(x_1,\ldots,x_n),\ldots,p^{(m)}(x_1,\ldots,x_n)) P=(y1​,y2​,…,ym​)=(p(1)(x1​,…,xn​),…,p(m)(x1​,…,xn​))

MPKC的困难问题定义如下：

MQ Problem : 给定m个多项式的结构，求解n个解 xˉi\bar{x}_{i}xˉi​ , 使得

p(1)(xˉ1,…,xˉn)=p(2)(xˉ1,…,xˉn)=…=p(m)(xˉ1,…,xˉn)=0p^{(1)}(\bar{x}_{1},\ldots,\bar{x}_{n})=p^{(2)}(\bar{x}_{1},\ldots,\bar{x}_{n})=\ldots =p^{(m)}(\bar{x}_{1},\ldots,\bar{x}_{n}) = 0 p(1)(xˉ1​,…,xˉn​)=p(2)(xˉ1​,…,xˉn​)=…=p(m)(xˉ1​,…,xˉn​)=0

一般性的MQ问题被证明在GF上是NP困难的。且当前不存在能够有效求解MQ问题的量子算法。

MPKC需要在一般性的MQ的基础上添加trapdoor来构造公私钥对。其核心想法是，利用一个可以很容易求解的 多项式映射（central map） F\mathcal{F}F , 通过在其前后复合可逆的Affine map S,TS, TS,T, 得到如下关键要素

  • 公钥 $ \mathcal{P}$ , 其为复合映射

  • 私钥为 F,S,T\mathcal{F}, S, TF,S,T
  • 解密/签名过程：给定m个结果(y1,…yn)(y_1,\ldots y_n)(y1​,…yn​), 对 P\mathcal{P}P 求逆得到一组解(x1,…,xn)(x_1,\ldots,x_n)(x1​,…,xn​)
  • 加密/验签过程：给定n个输入$(x_1, x_2,\ldots, x_n) $, 对多项式组求值

由于非线性映射不是一一映射，对m和n的选取有如下限制：加解密方案中要求 m≥nm\geq nm≥n,签名方案中要求 m≤nm\leq nm≤n 。对应于解密过程明文唯一，签名场景下签名存在。签名场景下可能存在其他的解，同一份消息可能有多个签名都能通过验签。
