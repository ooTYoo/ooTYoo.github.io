  • fuzzing介绍

    2020-03-04 |
    • { hacking }

    介绍

    自从90年代fuzzing技术发明以来，该技术成为发现软件漏洞最重要的半自动化技术。该技术广泛应用于发现web，数据库，软件包，内核等多个领域的安全研究中，相关工具种类繁多。文献The Art, Science, and Engineering of Fuzzing: A Survey给出了该领域相关术语和工具分类及关键技术的很好总结，本文是该文献的摘录。作为fuzzing技术入门的参考沙盘地图。

    术语定义

    • PUT: Program Under Test
    • fuzzing: fuzzing指使用特定的input参数运行PUT，这些参数通常来自于PUT运行的参数空间中，但又不同于PUT正常逻辑功能允许的参数子空间。
    • fuzzing test: fuzzing测试指利用fuzzing技术来测试PUT运行是否存在“安全问题”（violation of certain security policy）
    • fuzzer：fuzzer指对PUT进行fuzzing测试的程序
    • Fuzz Campaign：指在特定的fuzzer上测试PUT是否破坏特定security policy，其目标为发现bug(如crash PUT的特定testcase)
    • bug oracle：bug oracle指特定的程序逻辑，用于判断testcase是否为bug（破坏security policy）。bug oracle通常整合在fuzzer中。
    • fuzz algorithm： fuzzer中用到的算法
    • fuzz Configuration：指fuzz 算法中一组完备的参数集，该参数集可完整描述/控制 fuzz算法的内部状态。该参数集可能是固定的，也有可能随着fuzzing的进行而变动。
    • fuzz seed pool: fuzzer使用到的一组初始configuration集合。

    fuzzing 测试模型

    可以将fuzzing测试抽象为以下过程。

    fuzzing测试指在有限的时间$t_{limit}$内，通过输入的一组configuration集合$\mathbb{C}$，得到一组bug集合$\mathbb{B}$。该过程包含以下关键组件：

    • $Preprocess(\mathbb{C})\rightarrow \mathbb{C}$, 在fuzz campaign开始前的前处理过程。该过程可能进行的操作如对代码进行插桩（code instrumentation），对seed pool进行裁剪（seed trim），设置测试执行时间（timing）等，在一些fuzzer中，也可能不存在该步骤。
    • $Continue(\mathbb{C}) \rightarrow {True,False}$, 根据当前参数决定是否继续执行fuzzing测试
    • $Schedule(\mathbb{C},t_{elapsed},t_{limit})\rightarrow conf$, 根据当前参数和时间，参数下次迭代中需要使用的参数构型conf
    • $InputGen(conf)\rightarrow tcs$, 根据当前conf参数当前迭代对应PUT输入的testcase集合tcs。
    • $InputEval(conf,tcs,O_{bug})\rightarrow \mathbb{B}’,\text{execinfos}$, 根据当前conf和fuzz oracle，运行当前tcs，兵判断结果是否为bug，并收集对应的执行信息execinfos。
    • $ConfUpdate(\mathbb{C},conf,\text{execinfos})$，根据本次迭代的运行结果状态和conf更新fuzz configuration。

    fuzzer工具分类

    根据fuzzer每次迭代过程中获取的PUT语义信息（semantics）的粒度，可以将fuzzer分为以下三类

    • Black-box，又称为IO-driven或data-driven类型的fuzzer，将PUT做黑盒处理，仅分析输入输出，得到的信息较为粗略，fuzzer实现快，运行速率高，但相对的发现bug的效率低。
    • White-box，通过分析PUT内部细节信息来生成testcase，先谷底覆盖率高，发现问题可能性更高，但对PUT内部信息分析过程可能会很复杂，如动态符号执行（DES，Dynamic Symbolic Execution），约束求解 concolic （concrete + symbolic），Dynamic instrumentation，及SMT（Satisfiability modulo theories）等复杂技术。
    • Gray-box，通过轻量级的静态/动态信息分析，提升fuzz测试的效率。如利用覆盖率来（coverage）指导fuzzing测试的AFL

    Preprocess

    常见的preprocess如代码插桩，seed pool选择与优化，生成fuzzing所需的driver等。

    代码插桩可以反馈出PUT执行的内部状态信息，用于指导生成有效的testcase。插桩可以分析静态或动态的。静态插桩发生在代码执行前，通常是编译或链接阶段，overhead较小。动态插桩发生在运行中，可以fuzz到动态链接的库，以及动态加载的内容和内存，对执行效率损失较大，overhead大。另外还有binary level的插桩工具。

    为提升fuzz速度，甚至可以指fuzz代码执行过程的“局部”，例如将代码加载部分作为进程镜像（类似于Android中Zygote），该技术称为“in-memory fuzz”,镜像加载过程也可以放在preprocess中进行。该技术可显著提升fuzz运行速度，但一些发现的bug内内存执行的操作序列相关及相关side effect，bug会难以复现和定位分析。该技术在内核fuzz（in-memory API
    fuzzing）和AFL中都有应用。

    preprocess还可能包含进程调度等，用于fuzz进程并发中的race condition问题。

    正交且完备的seed pool可以有效提升fuzzing效率。如何优化seed pool的问题称为seed selection problem。一种常见的执行原则是选择一组最小种子，可以达到最大覆盖率，该中级集合称为minset。研究表明覆盖率提升1%，发现bug的概率提升0.92%。另外一种方式是在preprocess或者confupdate阶段对种子进行裁剪（Seed Trimming）。

    对于一些无法直接fuzz的PUT，需要在该阶段手动开发driver，如fuzz的对象为lib库函数，内核系统调用，IoT中的通信服务或手机app等。

    Scheduling

    Scheduling指从fuzz configuration中产生下次迭代使用到的配置。很多fuzzer的核心技术在于高效的scheduling算法。该算法要解决的核心问题（FCS，Fuzz Configuration Scheduling）是，如何通过分析现有的信息，产生更容易触发bug的配置。黑盒FCS中唯一可用的信息是当前输出，在灰盒测试中，可用的信息如coverage等。

    Input Generation

    Input Generation方法大致可用分为两种：

    • model less，mutation-based fuzzers，基于变异的fuzzer通过对seed的改变产生新的testcase，例如进行bit翻转。该类别对先验知识依赖较少，但在何处变异及如何变异是其需要解决的核心问题。
    • model-based，generation-based fuzzers，基于生成的fuzzer使用于PUT对输入有合法检查的场景，如magic number，CRC等，该场景下，随机的输入很容易本这些pre-check过滤，导致效率低下，如对于一些协议的fuzz测试中。该类别中，模型通常是公开的。未公开的模型需要进行复杂的模型分析或推测，该部分难度较大，可以与当前流行的深度学习DNN，LSTM等技术相互结合。

    Input Evaluation

    虽然该部分是最直接的，但该部分如何定义bug oracle，以及如何优化执行效率也是重要的研究领域。最常见的bug oracle为异常信号，如segmentation fault。该种判断标准无法覆盖全部的异常种类。该部分的演进出需要辅助工具（sanitizers），如Address Sanitizer，Memory Sanitizer等。执行优化部分如in-memory fuzz等。

    Triage指分析和报告触发bug oracle的测试例，其中包括去重，排序，优化等方面。

    Configuration Update

    该部分通常依赖于PUT执行信息，因此在黑盒fuzzing中较少修改配置，灰盒中，可以利用覆盖率对seed pool进行更新。

    启动调试

    gdb调试需要使用可执行文件中的符号表等调试信息。 编译时加上-g参数，保留调试信息，否则不能使用GDB进行调试。

    如果没有调试信息，gdb file指令会提示no debugging symbols found。

    gdb还可以用于调试核心转储文件，当程序core dump时，可能会产生core文件，它能够很大程序帮助我们定位问题。但前提是系统没有限制core文件的产生。可以使用命令limit -c查看，如果结果是0，程序core dump了也不会有core文件留下。需要使用如下命令进行设置unlimit-c unlimied。使用如下命令调试core文件
    gdb 程序文件名 core文件名

    常用指令有：

    • run/r 运行程序，可以带着程序运行所需要的参数
    • continue/c，继续运行直到下个断点
    • next/n，单步跟踪程序，当遇到函数调用时，也不进入此函数体；此命令同 step 的主要区别是，step 遇到用户自定义的函数，将步进到函数中去运行，而 next 则直接调用函数，不会进入到函数体内
    • step/s，单步调试，如果有函数调用，则进入函数；与命令n不同，n是不进入调用的函数的
    • until/u，当你厌倦了在一个循环体内单步跟踪时，这个命令可以运行程序直到退出循环体。
    • until+行号： 运行至某行，不仅仅用来跳出循环
    • finish： 运行程序，直到当前函数完成返回，并打印函数返回时的堆栈地址和返回值及参数值等信息。
    • call 函数（参数)：调用程序中可见的函数，并传递“参数”，如：call gdb_test(55)
    • quit/q，退出gdb

    断点相关指令

    • break n /b n，在第n行处设置断点（可以带上代码路径和代码名称： b OAGUPDATE.cpp:578）
    • b fn1 if a＞b，条件断点设置
    • break func/b func，在函数func()的入口处设置断点，如：break cb_button
    • delete n，删除第n个断点
    • disable 断点号n，暂停第n个断点
    • enable 断点号n，开启第n个断点
    • clear 行号n：清除第n行的断点
    • info b （info breakpoints），显示当前程序的断点设置情况
    • delete breakpoints：清除所有断点

    查看源代码

    • list /l ，其作用就是列出程序的源代码，默认每次显示10行。
    • list 行号，将显示当前文件以“行号”为中心的前后10行代码，如：list 12
    • list 函数名，将显示“函数名”所在函数的源代码，如：list main
    • list ，将接着上一次 list 命令的，输出下边的内容

    打印表达式

    print 表达式， 其中“表达式”可以是任何当前正在被测试程序的有效表达式，比如当前正在调试C语言的程序，那么“表达式”可以是任何C语言的有效表达式，包括数字，变量甚至是函数调用。

    • print a，将显示整数 a 的值
    • print ++a，将把 a 中的值加1,并显示出来
    • print name，将显示字符串 name 的值
    • print gdb_test(22)：将以整数22作为参数调用 gdb_test() 函数
    • print gdb_test(a)：将以变量 a 作为参数调用 gdb_test() 函数

    display 表达式，在单步运行时将非常有用，使用display命令设置一个表达式后，它将在每次单步进行指令后，紧接着输出被设置的表达式及值。如：

    • display awatch 表达式，设置一个监视点，一旦被监视的“表达式”的值改变，gdb将强行终止正在被调试的程序。
    • watch a
    • whatis，查询变量或函数
    • info function： 查询函数扩展info
    • locals： 显示当前堆栈页的所有变量

    查询运行信息

    • where/bt ，当前运行的堆栈列表
    • bt backtrace 显示当前调用堆栈
    • up/down 改变堆栈显示的深度
    • set args 参数，指定运行时的参数
    • show args，查看设置好的参数
    • info program，来查看程序的是否在运行，进程号，被暂停的原因
