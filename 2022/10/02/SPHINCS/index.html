SPHINCS

发表于 2022-10-02 更新于 2022-10-19

SPHINCS是基于hash的后量子签名算法，该算法在NIST第三轮中胜出。该算法和XMSS算法最大的不同在于其是stateless的，算法中无须维护可用密钥对的指标。其算法原理和基本组件和XMSS高度相同。本文中对该算法作个总结性介绍。

0x1 基本组件

SPHINCS算法是基于hash单向性的，其算法包括如下基本组件

  1. tweakable hash 算法，在标准的hash算法基础上，添加了key和消息掩码
  2. hash address scheme，标准中定义了5中不同的地址类型、
  3. WOTS+ 单次签名算法
  4. XMSS Merkel tree
  5. Hyper Tree
  6. FORS， Forestt Of Random Set

tweakable hash算法在单向函数中增加了key和掩码，提升了SPHINCS算法的安全行，也是WOTS+中加号的来源。对应的函数为

Tl:Bn×B32×Bln→Bnmd←Tl(PK.seed,ADRS,Mesg)T_l :\mathbb{B}^n \times \mathbb{B}^{32} \times \mathbb{B}^{ln} \rightarrow \mathbb{B}^n \\ md \leftarrow T_l(PK.seed, ADRS, Mesg) Tl​:Bn×B32×Bln→Bnmd←Tl​(PK.seed,ADRS,Mesg)

SPHINCS中的32Byte地址结构定义和XMSS具有较大不同，SPHINCS定义了五种不同类型的地址

  1. WOTS_HASH，用于WOTS+的hash中
  2. WOTS_PK，用于WOTS+公钥的压缩算符中
  3. TREE，用于Merkel Tree的构造中
  4. FORS_TREE，用于FORS Tree的构造中
  5. FORS_ROOTS，FORS Tree的root构造中

WOTS+的构造过程和XMSS相同，使用SK.seed 通过 PRF(SK.seed, ADRS) 产生 len 个 n-Byte的随机序列，之后使用单项函数迭代w-1次产生公钥。和XMSS中不同的是公钥的压缩方式。在XMSS中，len 个公钥使用L-Tree 生成一个n-byte公钥，在SPHINCS中，使用tweakable hash一次产生pk=T_len(PK.seed, wotspkADRS, wotspk[0:len-1]) 。

WOTS+的签名包含了len 个n-byte的hash计算结果。在SPHINCS中WOTS+用于在Hypter-Tree中，对于layer-0的root做签名。

Merkel Tree组件中，构建一颗高度为h 的平衡二叉树，共有 2h2^h2h 个叶节点，每个节点对应一个压缩后的WOTS+公钥。树的根节点作为XMSS的公钥。

XMSS的签名包含1个n-byte的WOTS+签名，以及由h个n-byte 串组成的Auth_path。共 (h+len)n(h+len)n(h+len)n Byte。验签过程需重现root节点的计算过程。

Hyper-Tree(HT)高度为 h′h'h′， 包含了 ddd 层，每层包含一系列高度为 h′/dh'/dh′/d 的Merkerl Tree，这些子树具备相同的密码学参数n,w 。SPHINCS中HT的叶结点为FORS Tree的公钥。layer-0为FORS-Tree，其上均为XMSS-Tree，layer-(n-1) 用于对其 layer-n 中子树的root进行WOTS+签名，所有签名中包含了d个WOTS+签名，总长度为 $d(h’/d + len)n $ Byte。

0x2 FORS

SPHINCS使用FORS来对实际消息做签名，而非XMSS中的WOTS+，这使得SPHINCS无须维护密钥对的使用状态。FORS是一颗高度为a+1的树，如下图所示

该树包含k个高度为a的平衡二叉子树，每颗子树有 t=2at=2^at=2a 个叶子节点，每个叶子节点为n-byte串，作为私钥使用。子树中中间节点的够作方法和Merkel-Tree类似，n个子树的根节点通过tweakable hash生存FORS的公钥。

FORSsk[i]=PRF(SK.seed,ADRS)FORSPK=TK(PK.seed,forspkADRS,root[0:k−1])FORS_{sk}[i] = PRF(SK.seed, ADRS)\\ FORS_{PK} = T_{K}(PK.seed, forspkADRS, root[0:k-1]) FORSsk​[i]=PRF(SK.seed,ADRS)FORSPK​=TK​(PK.seed,forspkADRS,root[0:k−1])

使用FORS对消息作签名时，需要先将消息M压缩后切割为k个片段，每个片段为a bit，将每个片段转化为数字，使用该数字作为指标来选择对应子树中的叶子节点。同时还需要为每个子树产生对应的auth_path, 包含a个n-byte串。共 k(a+1)nk(a+1)nk(a+1)n Byte。

回顾XMSS中，签名时也是将消息切片，每个切片转化为数字后，数字作为WOTS Chain中迭代次数的指示。在FORS中，签名方式只是个选择的过程，且相同的消息摘要对应相同的叶子节点，所以无需维护密钥对的使用状态。

0x4 算法优缺点

算法的优点包括：

  1. 良好的侧信道防御能力
  2. 极少的安全假设，只以来于单向函数的单向性
  3. 安全性分析较简单
  4. 复用了大量现有的密码学组建，方便构建

缺点：

  1. 调用大量的单项函数计算，性能慢
  2. 签名尺寸太大

在实际实现时，可以通过空间换性能对第一个缺点做缓解，但第二个缺点是制约SPHINCS+被广泛使用的关键因素。也是NIST发起第四轮评审的主要考虑，在第四轮中，主要是基于编码的密码算法了。
