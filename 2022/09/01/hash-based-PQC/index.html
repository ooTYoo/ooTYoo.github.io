0x1 Lamport-Diffie 单次签名方案

Lamport-Diffie one-time sigature sheme(LD-OTS)方案中，包含单项函数f，密码学hash函数g，满足

f:{0,1}n→{0,1}n,g:{0,1}∗→{0,1}nf:\{0,1\}^n\rightarrow \{0,1\}^n, \quad g:\{0,1\}^*\rightarrow \{0,1\}^n f:{0,1}n→{0,1}n,g:{0,1}∗→{0,1}n

n为安全参数。

Key Gen

在LD-OTS方案中，公私钥对中，签名私钥key X为2n个随机选取的n bit串组成。

X={xn−1[0],xn−1[1],…,x0[0],x0[1]}∈R{0,1}(n,2n)X = \{x_{n-1}[0],x_{n-1}[1],\ldots,x_{0}[0],x_{0}[1]\}\in_R \{0,1\}^{(n,2n)} X={xn−1​[0],xn−1​[1],…,x0​[0],x0​[1]}∈R​{0,1}(n,2n)

签名公钥Y为2n个长度为n的bit串，其中每个元素为X中对应元素的单项函数计算结果

Y={yn−1[0],yn−1[1],…,y0[0],y0[1]}∈R{0,1}(n,2n),yi[j]=f(xi[j])Y = \{y_{n-1}[0],y_{n-1}[1],\ldots,y_{0}[0],y_{0}[1]\}\in_R \{0,1\}^{(n,2n)},\quad y_i[j] = f(x_i[j]) Y={yn−1​[0],yn−1​[1],…,y0​[0],y0​[1]}∈R​{0,1}(n,2n),yi​[j]=f(xi​[j])

整个keygen过程需要随机取样2n个长度n的bit串，并计算2n次单项函数 f(x)f(x)f(x)。

Signature

对某个文件 M∈{0,1}∗M\in\{0,1\}^*M∈{0,1}∗ 签名计算过此如下

σ=(xn−1[dn−1],…,x0[d0])∈{0,1}(n,n),d=g(M)\sigma = (x_{n-1}[d_{n-1}],\ldots, x_0[d_0])\in \{0,1\}^{(n,n)},\quad d=g(M) σ=(xn−1​[dn−1​],…,x0​[d0​])∈{0,1}(n,n),d=g(M)

Verification

为验证签名 σ=(σn−1,…,σ0)\sigma = (\sigma_{n-1},\ldots,\sigma_{0})σ=(σn−1​,…,σ0​), 验签者只需判断

(f(σn−1,…,f(σ0))==(yn−1[dn−1],…,y0[d0])(f(\sigma_{n-1},\ldots,f(\sigma_{0})) == (y_{n-1}[d_{n-1}],\ldots, y_0[d_0]) (f(σn−1​,…,f(σ0​))==(yn−1​[dn−1​],…,y0​[d0​])

是否成立即可。

该方案中，签名的公私钥对只能使用1次。如果多次使用，每次使用都会泄漏私钥X的部分信息，攻击者收集到很多X的片段后，可以获得X，进而伪造任何签名。而只使用一次，攻击者如果先要破解该签名，需要破解单向函数 fff。故 OTS方案的安全星依赖于单项函数的单向性。

LD-OTS方案签名验签的计算复杂性均较低。但存在以下问题

  • 公私钥对长度均较长
  • 密钥只能使用一次，效率较低。

0x2 Winternitz单次签名方案

Winternitz one-time signature scheme (W-OTS)方案将签名在空间上的复杂性转化为计算单项函数的计算复杂性。该方案的好处是可以缩短签名的长度。该方案可以看作是Merkle Tree方案的设计原型。

W-OTS方案的参数为w，以及由w计算得到的 t1,t2,tt_1,t_2,tt1​,t2​,t

t1=⌈nw⌉,t2=⌈⌊log2t1+w+1w⌉,t=t1+t2t_1 = \lceil\frac{n}{w}\rceil, \quad t_2= \lceil\frac{\lfloor log_2{t_1}+w+1}{w}\rceil,\quad t=t_1+t_2 t1​=⌈wn​⌉,t2​=⌈w⌊log2​t1​+w+1​⌉,t=t1​+t2​

Key Gen

在W-OTS方案中，公私钥对中，签名私钥key X为t个随机选取的n bit串组成。

X={xt−1[0],…,x0}∈R{0,1}(n,t)X = \{x_{t-1}[0],\ldots,x_{0}\}\in_R \{0,1\}^{(n,t)} X={xt−1​[0],…,x0​}∈R​{0,1}(n,t)

签名公钥Y为t个长度为n的bit串，其中每个元素为X中对应元素的单项函数计算结果, 不同于LD-OTS方案，此处的单向函数计算过程迭代2w−12^w-12w−1 次。

Y={yn−1,…,y0},yi[j]=f2w−1(xi[j])Y = \{y_{n-1},\ldots,y_{0}\},\quad y_i[j] = f^{2^w-1}(x_i[j]) Y={yn−1​,…,y0​},yi​[j]=f2w−1(xi​[j])

整个keygen过程需要随机取样t个长度n的bit串，并计算t(2w−1)t(2^w-1)t(2w−1) 次单项函数 f(x)f(x)f(x)。

Signature
对某个文件 M∈{0,1}∗M\in\{0,1\}^*M∈{0,1}∗ 签名计算过此如下

σ=(fbt−1(xt−1,…,fb0(x0)),d=g(M)\sigma = (f^{b_{t-1}}(x_{t-1},\ldots, f^{b_0}(x_0)),\quad d=g(M) σ=(fbt−1​(xt−1​,…,fb0​(x0​)),d=g(M)

Verification

为验证签名 σ=(σt−1,…,σ0)\sigma = (\sigma_{t-1},\ldots,\sigma_{0})σ=(σt−1​,…,σ0​), 验签者只需判断

(f2w−1−bt−1(σt−1),…,f2w−1−b0(σ0))==(yt−1,…,y0)(f^{2^w-1-b_{t-1}}(\sigma_{t-1}),\ldots, f^{2^w-1-b_0}(\sigma_0)) == (y_{t-1},\ldots, y_0) (f2w−1−bt−1​(σt−1​),…,f2w−1−b0​(σ0​))==(yt−1​,…,y0​)

是否成立即可。

签名和验签的过程相当于将原本 2w−12^w-12w−1 此的单向函数计算过程，根据文件hash，分成两步完成该计算过程，第一步作为签名，第二步作为验签。

W-OTS的安全行仍然是单向函数的单向性。且公私钥对依然只能够使用一次。

0x3 Merkle’s Tree Scheme

OTS方案中，每个公私钥对只能使用一次，如果使用OTS作为签名方案，需要派发大量的公钥。1979年Merkle提出了一种方案来解决大量公钥的问题，该方案中的核心组件是称为Merkle‘s Tree的结构。

Merkle‘s Tree是完全二叉树。假设现在有 2h2^h2h 组OTS公私钥对，其Merkle‘s Tree性质如下

  • Merkle‘s Tree是一颗高度为h的二叉树，
  • 叶子节点对应于每个OTS公钥y的hash
  • 每个中间节点为其两个字节点串联后的hash， node=hash(leltChild∣∣rightChild)node = hash(leltChild || rightChild)node=hash(leltChild∣∣rightChild)
  • Merkle’s Tree的根节点为派发的公钥。

该方案下的签名包含两部分内容

  • 使用编号为s的OTS私钥进行签名，得到 σOTS\sigma_{OTS}σOTS​
  • 提供从s对应叶子节点到Merkle’s Tree 根节点路径上，每个节点的兄弟节点 (a0,…,aH−1)(a_0,\ldots,a_{H-1})(a0​,…,aH−1​), 该信息称为Authentication Path。
  • 最终签名为S=(s,Ys,σOTS,(a0,…,aH−1)))S=(s, Y_s, \sigma_{OTS}, (a_0,\ldots,a_{H-1})))S=(s,Ys​,σOTS​,(a0​,…,aH−1​)))

在签名校验过程中，校验者

  • 使用YsY_sYs​验证OTS签名方案的合法性
  • 利用Ys,(a0,…,aH−1))Y_s,(a_0,\ldots,a_{H-1}))Ys​,(a0​,…,aH−1​)) 信息计算根节点的hash，并和公钥进行比对，进而校验YsY_sYs​的合法性。

0x4 MSS方案

基于hash构建的签名方案具备以下特点

  • 依赖于较少的安全假设
  • 具备抵抗量子计算机攻击的能力
  • 实现方式较为简单
  • 具备自然的抗SCA分析能力等

基于hash的PQC算法可以分为两大类

  1. Stateful的，需要在使用中维护状态信息，如XMSS, LMS
  2. Stateless的，如NIST PQC候选算法SPHINCS

RFC8391定义的基于hash的eXtended Merkle Signature Scheme(XMSS)方案，该方案使用了W-OTS的改进版本W-OTS+，以及支持Multi-Tree的变种XMSS^MT, 可以在底层hash算法抗碰撞攻击的特性不足时，仍然维持足够的安全性。

0x4.1 W-OTS+方案

WOTS+在WOTS的基础上，将底层的单项函数替换为代key的hash算法，并且将hash计算的输入进行了加掩码保护，hash key和掩码通过PRF(pseudo Random function) 产生，PRF的输入包括

  1. n-byte的seed
  2. 32Byte的Address结构

在后续的WOTS签名和验签的基础上，都添加了基于掩码的保护逻辑

0x4.2 L-Tree

RFC定义了对多个W-OTS+公钥进行压缩时使用的L-Tree算法，该算法可以将多个(个数不要求为2k2^k2k)个chain末端的pk压缩为单个L-Tree的根节点。并使用L-Tree构建Merkle Tree。

0x4.3 Multi-Tree

XMSS方案中需要保障每个W-OTS+公私钥对只使用一次，在签名方案中需要维护当前可用的签名密钥对信息。当使用单个Merkle Tree支持多个公私钥对时，Merkle-Tree根节点的计算需要所有密钥对的多次hash计算，这给密钥派生操作带来了巨大的负担。Multi-Tree方案可以缓解该问题。

所谓的Multi-Tree，是使用多个具备相同参数（高度，w位宽等）的Merkle-Tree组成的hyper Tree，该hyperTree的叶子节点为使用WOTS+方案对对应Merkle-Tree根节点的签名。HyperTree简化了公钥真实行的校验所需的hash计算量。
